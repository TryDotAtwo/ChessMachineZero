# ChessMachineZero: Percepta-style chess transformer architecture

Version: 2.0
Status: Codex-ready design specification
Goal: build a chess transformer that learns only from self-play while using a Percepta-style internal computation architecture.

---

## 1. Hard architectural rule

The system must not be organized as an external tree-search system around a policy/value model.

The system must be organized as an executable trace machine:

```text
input board + internal program + stochastic seed
→ transformer executes chess computation inside its own decoding loop
→ transformer emits append-only trace tokens
→ trace computes legal moves, board transitions, terminal rules, move ranking, and move commitment
→ self-play outcome updates trainable decision components
```

No human PGN training.
No engine labels.
No tablebase labels.
No opening book.
No handcrafted material/piece-square evaluation function.
No external evaluator.
No external move selector at inference time.

Allowed:

```text
chess rules
legal-move verification during development
terminal-result verification during development
randomness for exploration
self-play game outcomes
trace consistency losses derived from rules
```

## 2. Percepta pattern translated to chess

Percepta pattern:

```text
program tokens
+ transformer-embedded interpreter
+ append-only execution trace
+ small fixed number of lookbacks per step
+ 2D hardmax/top-k attention
+ convex-hull-backed fast KV lookup
```

ChessMachineZero counterpart:

```text
chess program tokens
+ ChessMachineVM inside transformer weights
+ append-only board/game/move-selection trace
+ explicit memory records for squares, registers, stack, candidates, clocks, hashes
+ 2D attention heads for latest-write and stack/candidate retrieval
+ HullKVCache for long self-play and internal game-tree computation traces
```

The Sudoku example in the Percepta article should be treated as the closest design analogy:

```text
Sudoku in Percepta:
  compiled solver executes inside transformer
  constraint propagation and backtracking are trace operations
  correctness comes from executing a correct algorithm, not from guessing a grid

ChessMachineZero:
  compiled chess VM executes inside transformer
  legal move generation and make-move logic are trace operations
  move choice comes from trainable self-play decision modules, not from an external engine
```

The strategic part is learned.
The rule-execution part is deterministic.

## 4. Core technical object: append-only chess trace

Mutable chess state must be represented as immutable trace records.

### 4.1 Trace record format

Use fixed-width token packets for implementation simplicity.

```text
TracePacket = [OP, A0, A1, A2, A3, TAG, COMMIT]
```

Recommended token classes:

```text
OP      = operation token
A0..A3  = small integer / square / piece / move / pointer / score bucket fields
TAG     = phase tag or address-space tag
COMMIT  = line boundary and state-advance marker
```

Examples:

```text
WRITE_SQ      square piece ply version
READ_SQ       square target_register
WRITE_REG     register value version
PUSH_STACK    stack_id value stack_pointer
POP_STACK     stack_id stack_pointer
WRITE_CASTLE  castle_mask ply version
WRITE_EP      ep_file_or_none ply version
WRITE_CLOCK   halfmove_clock fullmove_number
WRITE_HASH    zobrist_hi zobrist_lo repetition_count
ATTACK_SET    color square attacked_bit
PIN_SET       square ray_direction pinned_bit
CHECK_SET     checking_piece_square check_type
CANDIDATE     move_id from to promotion
LEGAL_SET     move_id legal_bit reason_code
SCORE_SET     move_id score_bucket source_depth
SAMPLE_SET    seed temperature chosen_rank
COMMIT_MOVE   move_id from to promotion
TERMINAL_SET  result_code reason_code ply
HALT_GAME     result_code total_plies
```

### 4.2 Current state reconstruction

Current board is obtained by latest writes:

```text
piece(square)        = latest WRITE_SQ for square
castling_rights      = latest WRITE_CASTLE
current_ep_file      = latest WRITE_EP
halfmove_clock       = latest WRITE_CLOCK.halfmove_clock
fullmove_number      = latest WRITE_CLOCK.fullmove_number
position_hash_count  = latest WRITE_HASH.repetition_count for current hash
terminal_status      = latest TERMINAL_SET or none
```

This mirrors a computer memory model without mutable memory.
The trace only grows.
The transformer reads previous records through attention.

### 4.3 Lookup requirements

Each VM step should require a bounded number of conceptual lookbacks:

```text
latest board square write
latest register write
current stack top
current program instruction
previous candidate record
latest score for candidate
latest hash-count record
```

Dense attention is the correctness oracle.
HullKVCache is the optimized lookup path after equivalence tests pass.

## 5. Transformer architecture

### 5.1 Executor-compatible transformer

Use a standard causal transformer with head dimension exactly 2 for the executor path.

```python
class CMZMachineTransformer(nn.Module):
    def __init__(self, vocab_size, d_model=256, n_heads=128, n_layers=12, d_ff=1024):
        assert d_model % n_heads == 0
        assert d_model // n_heads == 2
        ...
```

Required properties:

```text
causal decoding
2D attention heads for executor layers
hardmax attention mode for deterministic VM lookup heads
k-sparse top-k mode for trainable ranking heads
gated feed-forward blocks
separate output heads for trace tokens, move-score tokens, outcome-baseline tokens
```

### 5.2 Layer partition

```text
Layers 0..L_rule-1:
  deterministic trace execution support
  latest-write retrieval
  stack/register manipulation
  legal-rule computation

Layers L_rule..L_rank-1:
  candidate aggregation
  board summary construction
  move tactical feature extraction from trace

Layers L_rank..L-1:
  trainable move ranking
  stochastic action selection
  outcome baseline
```

Initial implementation can use one shared transformer stack.
Later implementation can freeze rule-execution layers and train ranker layers/adapters.

### 5.3 Attention modes

```text
DenseHardmax2D:
  exact brute-force hardmax over prefix
  used in tests and early training

HullHardmax2D:
  convex-hull support query for 2D keys
  returns same argmax as DenseHardmax2D
  used for long traces

NestedHullTopK2D:
  retrieves top-k keys via nested convex hulls
  applies softmax only over retrieved keys
  used for ranking heads when exact softmax is unnecessary
```

Acceptance condition:

```text
for every tested prefix and query:
  DenseHardmax2D(query, prefix_keys) == HullHardmax2D(query, prefix_keys)

for top-k mode:
  set(DenseTopK2D(query, prefix_keys, k)) == set(NestedHullTopK2D(query, prefix_keys, k))
```

## 6. ChessMachineVM

ChessMachineVM is the internal program executed by the transformer.

### 6.1 Recommended implementation path

Prefer a direct chess bytecode VM before a full WebAssembly interpreter.

Reason:

```text
full WASM interpreter = closer to Percepta demo, higher engineering cost
chess bytecode VM     = same architectural principle, much smaller surface area
```

Keep a WASM-compatible extension point:

```text
src/chess_machine_zero/vm/wasm_bridge.py
```

Do not implement the WASM bridge until the direct chess bytecode VM passes trace tests.

### 6.2 VM instruction set

Minimum instruction groups:

## 7. Move representation

Use compact move packets rather than a flat policy head as the primary interface.

```text
MovePacket = [FROM_SQ, TO_SQ, PROMO, FLAGS]
```

Fields:

```text
FROM_SQ: 0..63
TO_SQ:   0..63
PROMO:   none | N | B | R | Q
FLAGS:   quiet | capture | ep | castle | promotion | check | mate_candidate
```

Optional internal `move_id`:

```text
move_id = from * 64 * 5 + to * 5 + promo_id
```

Do not expose illegal moves as selectable actions.
The VM enumerates candidates, filters legality, and only legal candidate records are sent to the ranker.

## 8. Move selection without external search wrapper

The model selects moves through a trace-executed program.

### 8.1 Minimal move-selection program

```text
SELECT_MOVE(board, seed, temperature):
  legal_moves = EXECUTE_LEGAL_GENERATOR(board)
  for move in legal_moves:
      child_trace = EXECUTE_MAKE_MOVE(board, move)
      score[move] = CALL_MOVE_RANKER(board_trace, move_trace, child_trace)
  chosen = SAMPLE_OR_ARGMAX(score, seed, temperature)
  EMIT COMMIT_MOVE(chosen)
```

### 8.2 Optional internal lookahead program

The Percepta architecture is useful because deterministic algorithms can run inside the transformer.
Use an internal depth-limited game-tree program after the minimal selector works.

```text
TRACE_NEGAMAX(board, depth):
  if terminal(board):
      return terminal_score(board)
  if depth == 0:
      return CALL_OUTCOME_BASELINE(board)
  best = -inf
  legal_moves = EXECUTE_LEGAL_GENERATOR(board)
  for move in legal_moves:
      child = EXECUTE_MAKE_MOVE(board, move)
      score = -TRACE_NEGAMAX(child, depth - 1)
      best = max(best, score)
  return best
```

Important restriction:

```text
TRACE_NEGAMAX is an internal program executed as trace tokens.
TRACE_NEGAMAX must not call an external chess engine.
TRACE_NEGAMAX leaf values come from CMZOutcomeBaseline trained only from self-play outcomes.
```

This gives the architecture a direct analogue of Sudoku backtracking:

```text
Sudoku solver: recursive constraint/backtracking trace
Chess selector: recursive legal-game-tree trace with learned leaf baseline
```

## 9. Self-play training

### 9.1 Self-play game generation

```text
for game_id in batch:
    initialize legal starting position
    trace = INIT_GAME_TRACE(position, seed)
    while not terminal:
        trace += SELECT_MOVE_TRACE(position, seed, temperature_schedule)
        position = APPLY_COMMITTED_MOVE_FOR_HOST_IO(position, move)
        trace += TERMINAL_CHECK_TRACE(position)
    assign final result z ∈ {-1, 0, +1} to every decision record from side-to-move perspective
    store trace windows and decision metadata
```

The host-side `APPLY_COMMITTED_MOVE_FOR_HOST_IO` is only an I/O adapter during training.
The model-produced trace must contain enough information to reconstruct the same position independently.

### 9.2 Replay records

```text
ReplayRecord:
  game_id
  ply
  side_to_move
  input_trace_window
  decision_trace_span
  legal_candidate_span
  chosen_move_packet
  logprob_chosen
  entropy
  temperature
  final_outcome_from_side_to_move
  trace_hash_before
  trace_hash_after
  rules_version
  model_checkpoint_id
  stochastic_seed
```

### 9.3 Loss functions

Total loss:

```text
L_total =
    λ_trace      * L_trace_next_token
  + λ_rule       * L_rule_invariant
  + λ_pg         * L_selfplay_policy_gradient
  + λ_base       * L_outcome_baseline
  + λ_entropy    * L_entropy_regularizer
  + λ_consistency* L_trace_consistency
  + λ_compress   * L_internal_trace_distillation
```

Definitions:

```text
L_trace_next_token:
  cross-entropy over deterministic VM trace tokens
  target source: rules-generated trace oracle only
  purpose: teach/verify exact execution mechanics, not strategy

L_rule_invariant:
  penalties for impossible trace states
  examples: two kings missing, own king left in check after LEGAL_SET, illegal ep square, invalid castle right

L_selfplay_policy_gradient:
  - advantage * log probability of chosen move
  advantage = final_outcome - outcome_baseline(input_trace)

L_outcome_baseline:
  regression/classification loss predicting final result from trace state
  target source: self-play game result only

L_entropy_regularizer:
  maintains exploration early in training

L_trace_consistency:
  trace hash, board reconstruction, make/unmake, terminal-state consistency

L_internal_trace_distillation:
  optional self-distillation from deeper internal trace program to shallower move-ranker
  target source: current model's own longer computation, not external data
```

### 9.4 Training stages

```text
Stage A: VM trace oracle
  implement host ChessMachineVM interpreter
  generate deterministic traces from chess rules
  verify exact legal moves against python-chess

Stage B: executor imitation / compilation
  train or compile CMZTraceExecutor to reproduce VM trace transitions
  dense attention only
  no strategic training yet

Stage C: legal self-play with ranker
  use trace VM for legal generation
  train CMZMoveRanker from self-play outcomes
  stochastic move selection
  no internal game-tree program yet

Stage D: internal trace lookahead
  add TRACE_NEGAMAX depth 1, then depth 2+
  leaf baseline trained from self-play
  trace length grows; dense attention remains correctness oracle

Stage E: HullKVCache
  enable 2D hardmax/top-k fast path
  require dense-vs-hull equivalence tests

Stage F: full trace-game generation
  reduce host involvement to I/O and auditing
  model emits complete game traces or long chunks
```

```text
starting position legal moves = 20
castling legality: blocked, attacked-through, moved rook, moved king
promotion: all four promotion pieces
underpromotion: N/B/R legal when legal
pinned piece cannot expose king
king cannot move into check
en-passant cannot expose king through discovered attack
checkmate terminal
stalemate terminal
threefold repetition detection
```

```text
trace reconstructs exact board after every committed ply
latest WRITE_SQ lookup equals host board square
legal candidate set equals rules oracle legal set

terminal trace agrees with host terminal result
trace hash deterministic across repeated runs with same seed
```

```text
CMZMachineTransformer output shape correct
trace-token head predicts next packet fields
hardmax attention returns exact dense argmax
HullKVCache returns exact dense hardmax result
```

### 11.4 Self-play

```text
no illegal move in 10k generated plies
game terminates by mate/stalemate/draw/adjudication cap
```

## 14. Design invariant summary

```text
ChessMachineZero is not a model that merely predicts the next chess move.
ChessMachineZero is a transformer-hosted chess machine.
The machine computes rules through an append-only trace.
The machine learns move choice from self-play outcomes.
The machine can later run internal deterministic lookahead as trace computation.
The optimized executor path uses 2D hardmax/top-k attention and HullKVCache.
```
